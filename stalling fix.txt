Find the hot process first. Open Help → Open Process Explorer while the stall is happening, or run code --status in a terminal. If extensionHost is the one spiking, an extension is usually the cause. VS Code’s own guidance is to retry with code --disable-extensions, then run Help: Troubleshoot Issue or Help: Start Extension Bisect to isolate the bad extension quickly.
Cut the repo down for watchers and search. In large workspaces, VS Code recommends excluding high-churn folders from file watching, and its search docs say big repos, lots of symlinks, and very long lines can make search slow or even hang. Also make sure search is still respecting ignore files/excludes.
{
  "files.watcherExclude": {
    "**/.git/objects/**": true,
    "**/.git/subtree-cache/**": true,
    "**/node_modules/**": true,
    "**/.next/**": true,
    "**/dist/**": true,
    "**/build/**": true,
    "**/coverage/**": true,
    "**/.venv/**": true,
    "**/target/**": true
  },
  "search.exclude": {
    "**/node_modules": true,
    "**/.next": true,
    "**/dist": true,
    "**/build": true,
    "**/coverage": true,
    "**/.venv": true,
    "**/target": true
  },
  "search.followSymlinks": false
}
If you’re on Linux/WSL and saw the “unable to watch for file changes” warning, raise inotify after adding excludes. VS Code documents ENOSPC as a file-watcher limit problem in large workspaces and suggests excluding large folders first, then increasing fs.inotify.max_user_watches if needed.
cat /proc/sys/fs/inotify/max_user_watches
echo fs.inotify.max_user_watches=524288 | sudo tee -a /etc/sysctl.conf
sudo sysctl --system
If the slowdown is specifically AI/agent edits: keep the scope tight. VS Code’s Copilot docs say fewer enabled tools means faster, more relevant responses; instruction files should stay concise; and agents are best for multi-file work, while inline chat is better for targeted in-place edits.
If Git operations are what stall, inspect Git logs instead of guessing. Open Git: Show Git Output, and if needed raise the log level to trace; the output includes the Git executable path, commands VS Code ran, timestamps, durations, and errors.

A couple of situational fixes:

On Windows/macOS with Dev Containers, bind-mounted source trees are slower; VS Code recommends storing the repo in the WSL 2 filesystem on Windows or using Clone Repository in Container Volume / named volumes for better performance.
In WSL / SSH / Containers, Git push/sync can hang if your SSH key has a passphrase; the docs suggest using a key without a passphrase, HTTPS, or running the Git command in the terminal.
